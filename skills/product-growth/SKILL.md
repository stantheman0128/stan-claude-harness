# Product Growth（Grower）

把已上線的產品，靠量測跟實驗往上推。核心紀律只有三條：**先量再做、一次一個實驗、用 organic 手段**。沒有數字就先去拿數字，不要憑感覺加功能。

這個 skill 補的是你整個 stack 唯一沒人管的一段：產品上線「之後」。上線前的需求/成功指標是 requirements-analyst，開發新功能是 guided-dev，這裡只管「東西已經在跑了，怎麼讓更多人用、用得更久」。

## 適用的產品面（你的實際戰場）

- Play Store 上架頁 + 留存（Notify+ 這類 Android app）
- Chrome Web Store 轉換 + 評分（你的擴充功能）
- 線上 web app 的啟用與回訪（course、portfolio、各 tracker）

不適用：還沒有真實用戶的東西（那是 Prototyper/Builder 的事，這裡沒數字可量）。

## 流程

### 1. 先反問現況，一次問一題

動手前先把這幾格填起來，缺哪格就問使用者（一次一題，不要一次丟一串）：

- 哪個產品、哪個面（商店頁 / app 內 / web）？
- 現在「最該動的那一個數字」是什麼？（別同時追五個）
- 這個數字現在多少、你覺得多少叫好？
- 主要進來的管道是什麼（搜尋 / 朋友轉 / 商店推薦 / 完全沒有）？
- 手上有什麼量測？（GA4 / Plausible / PostHog / 商店後台 / 完全沒有）

沒有量測是常見起點。那第一個「實驗」往往就是裝最輕量的量測，先看得到漏斗再說。

### 2. 用 AARRR 找破口

把使用者旅程拆成幾段，找「漏最兇」的那一段，只修那段：

- **取得 Acquisition**：曝光→安裝/註冊。商店頁的圖示、截圖、標題、描述、評分都在這。
- **啟用 Activation**：第一次用→真的體會到價值的那一刻（aha）。新手流程、空狀態、第一個成功動作。
- **留存 Retention**：D1 / D7 / D30 回訪。為什麼裝了不用、用一次就走。通常這段最值錢。
- **推薦 Referral**：用戶帶用戶。分享按鈕、邀請、自然口碑。
- 營收 Revenue：你多數專案沒有，跳過。

判準：往源頭找第一個明顯掉很多的階段。衝流量進來但留存破洞，是在漏桶裡加水。

### 3. 列實驗，用 ICE 挑一個

針對破口列幾個可做的改動，每個打三個分（1 到 10）：

- **Impact** 成功的話這個數字會動多少
- **Confidence** 你多有把握它會成功
- **Ease** 多好做

`ICE = (I + C + E)` 取最高的「一個」先做。一次一個，不然數字動了你不知道是誰動的。

### 4. 設計成可量測

實驗開跑前先講清楚：

- 量哪個數字、怎麼記（埋一個事件 / 看商店後台 / 手動拉一次 cohort）
- 成功門檻是多少（事先寫死，別事後找理由）
- 要不要 A/B：流量夠大才 A/B；流量小就 before/after 或分時段比，並老實標「樣本小、僅供參考」
- 跑多久才算數（別跑兩天就下結論）

### 5. 跑完判讀

數字有沒有超過雜訊動起來？三條路：**留**（成功，標準化）、**砍**（沒用，回復）、**改一版再試**。每輪記一行：假設、做了什麼、結果、下一步。

## Scope 守則（重要）

預設只走 organic 與 product-led：商店頁優化、留存、啟用、產品內的自然成長迴圈。**不要**自動跳到付費廣告、LTV:CAC、再行銷、成長黑客那套 VC-SaaS 劇本，除非使用者明講有預算要投放。對小 app / 擴充 / 個人專案，那套是廢動作。

## 反模式

- 沒有量測就開始加功能
- 追虛榮指標（只看安裝數不看留存）
- 同時跑一堆實驗，最後分不清誰有效
- 商店頁一次改五個地方，不知道是哪個讓轉換變好
